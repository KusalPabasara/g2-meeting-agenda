> BYPASS PATH (no model needed):

Raw GPS coordinates from Group 1 can go directly to Group 3 for live map rendering. They do not need to wait for our ETA prediction — they just plot the bus dot on the map. Our model adds the intelligence layer on top: “When will this bus arrive at your stop?”

> WHAT THIS MEANS:

Group 3 consumes two streams from us:
1. Pass-through GPS → Real-time bus position on map (no processing)
2. ETA predictions → “Bus arrives in 7 min” (processed through our model)
Both travel through Kafka topics but on separate channels.

> Building Bus Tracking
How to Build It Like Google Maps — Without Relying on Google
We cannot depend on paid third-party APIs in production. Here is how we build our own tracking and distance system from scratch.
tracking-strategy.md

> THE PROBLEM

Google Maps and Apple Maps APIs cost money at scale and create vendor lock-in. For a Sri Lanka public transport system, we need a free, self-hosted solution that we fully control.

> OUR SOLUTION: 4 OPEN-SOURCE LAYERS

01
Map Data — OpenStreetMap (OSM)
Download the full Sri Lanka road network from Geofabrik (free). This gives us every road, intersection, bus stop, and route geometry. Updated regularly by the community. We host it ourselves using a PostGIS database or flat files.
02
Routing Engine — OSRM / Valhalla
OSRM (Open Source Routing Machine) takes OSM data and gives us: shortest path between two points, driving distance, turn-by-turn directions, and ETA based on road speed limits. Self-hosted, free, fast. Drop-in replacement for Google Directions API. Valhalla is an alternative by Mapbox (also open source).
03
Bus Stop Distances — Haversine + Road Snap
Haversine formula gives straight-line distance between two GPS coordinates (good enough for nearby stops). For road distance, we use OSRM’s /route endpoint with the two stop coordinates — it returns actual driving distance along the road. We pre-compute all stop-to-stop distances as a staircase matrix and cache it.
04
Map Display — Leaflet.js + OSM Tiles
Leaflet.js is a free, open-source map library. Paired with free OSM tile servers, it replaces Google Maps UI completely. Group 3 uses Leaflet to render the map, plot bus positions, and show routes. No API key needed. Optionally self-host tiles with tileserver-gl for zero external dependency.
haversine-formula.py

> HAVERSINE: Calculate straight-line distance between two GPS points

import math

def haversine(lat1, lon1, lat2, lon2):
    R = 6371000  # Earth radius in metres
    phi1, phi2 = math.radians(lat1), math.radians(lat2)
    d_phi = math.radians(lat2 - lat1)
    d_lambda = math.radians(lon2 - lon1)
    a = math.sin(d_phi/2)**2 + \
        math.cos(phi1) * math.cos(phi2) * math.sin(d_lambda/2)**2
    return R * 2 * math.atan2(math.sqrt(a), math.sqrt(1-a))

# Example: Pettah to Fort Railway
dist = haversine(6.9355, 79.8510, 6.9344, 79.8428)
print(f"Distance: {dist:.0f} metres")  # ~880m

Kafka — Message Buffering & Streaming +

What it does: Kafka sits between Group 1’s GPS sensors and our processing pipeline. Every GPS ping lands in a Kafka topic (like a queue). Our consumers (Flink/Spark) read from it at their own speed. If our model is slow or down, the data is safe in Kafka.

Quick Start (Docker):

# docker-compose.yml
version: '3'
services:
  zookeeper:
    image: confluentinc/cp-zookeeper:7.5.0
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
  kafka:
    image: confluentinc/cp-kafka:7.5.0
    depends_on: [zookeeper]
    ports: ["9092:9092"]
    environment:
      KAFKA_BROKER_ID: 1
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1

Create our topics:

# Topic for raw GPS data from Group 1
kafka-topics --create --topic gps-raw \
  --bootstrap-server localhost:9092 \
  --partitions 3 --replication-factor 1

# Topic for ETA predictions (consumed by Group 3)
kafka-topics --create --topic eta-predictions \
  --bootstrap-server localhost:9092 \
  --partitions 3 --replication-factor 1

Python Producer (simulate GPS):

from kafka import KafkaProducer
import json, time

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode()
)

# Send a GPS ping
producer.send('gps-raw', {
    'bus_id': 'BUS042',
    'route_id': '138',
    'lat': 6.9271,
    'lng': 79.8612,
    'speed_kmh': 32.5,
    'ts': int(time.time() * 1000)
})
Spark — Batch Processing & Feature Engineering +

What it does: Spark processes large historical datasets — millions of past GPS records — to compute features for model training. Think: “average delay on route 138 at 8 AM on Mondays.”

Install:

pip install pyspark

Example — Compute average delay per route per hour:

from pyspark.sql import SparkSession
from pyspark.sql.functions import hour, avg, col

spark = SparkSession.builder.appName("G2-Features").getOrCreate()

# Load historical GPS data
df = spark.read.parquet("s3://g2-data/gps-history/")

# Feature: average delay by route and hour
features = (df
    .withColumn("hour", hour("timestamp"))
    .groupBy("route_id", "hour")
    .agg(avg("delay_minutes").alias("avg_delay"))
)

features.write.parquet("s3://g2-data/features/avg-delay/")

Run nightly as a batch job. The output becomes a lookup table for the ML model.

Flink — Real-Time Stream Processing +

What it does: Flink processes the live GPS stream in real time. As each GPS ping arrives from Kafka, Flink: validates it, snaps it to the nearest road, computes current speed/heading, and triggers the ETA model.

Why Flink over Spark Streaming? Flink has true event-time processing and lower latency (<100ms). For GPS data arriving every 5 seconds, Flink is the right choice.

Python (PyFlink) example:

from pyflink.datastream import StreamExecutionEnvironment
from pyflink.datastream.connectors.kafka import FlinkKafkaConsumer
from pyflink.common.serialization import SimpleStringSchema

env = StreamExecutionEnvironment.get_execution_environment()

# Read from Kafka topic
consumer = FlinkKafkaConsumer(
    topics='gps-raw',
    deserialization_schema=SimpleStringSchema(),
    properties={
        'bootstrap.servers': 'localhost:9092',
        'group.id': 'g2-flink-processor'
    }
)

stream = env.add_source(consumer)

# Process: validate, snap to road, compute features
processed = stream \
    .map(parse_gps_json) \
    .filter(is_valid_gps) \
    .map(snap_to_road) \
    .map(predict_eta)

env.execute("G2-GPS-Stream")
ML Model Training — scikit-learn + MLflow +

What it does: We train a model that predicts: “given this bus, this route, this time, this current position — how many minutes until it arrives at stop X?”

Training pipeline:

import pandas as pd
from sklearn.ensemble import GradientBoostingRegressor
from sklearn.model_selection import train_test_split
from sklearn.metrics import mean_absolute_error
import mlflow, mlflow.sklearn

# Load features computed by Spark
df = pd.read_parquet("features/training_data.parquet")

X = df[['route_id_enc', 'hour', 'day_of_week', 'is_rush',
         'speed_kmh', 'distance_to_stop', 'avg_hist_delay']]
y = df['actual_delay_minutes']

X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)

# Track with MLflow
mlflow.set_experiment("g2-eta-prediction")
with mlflow.start_run():
    model = GradientBoostingRegressor(
        n_estimators=200, max_depth=6, learning_rate=0.1
    )
    model.fit(X_train, y_train)

    preds = model.predict(X_test)
    mae = mean_absolute_error(y_test, preds)

    mlflow.log_param("model", "GradientBoosting")
    mlflow.log_param("n_estimators", 200)
    mlflow.log_metric("mae_minutes", mae)
    mlflow.sklearn.log_model(model, "eta-model")

    print(f"MAE: {mae:.1f} minutes")  # Target: <3 min
FastAPI — Serving Predictions +

What it does: Wraps our trained model in an HTTP endpoint. Group 3 sends a request, gets back an ETA in milliseconds.

# app.py
from fastapi import FastAPI, Query
import mlflow, pandas as pd

app = FastAPI(title="G2 ETA Prediction API")
model = mlflow.sklearn.load_model("runs:/<RUN_ID>/eta-model")

@app.get("/api/eta")
def predict_eta(
    route_id: str = Query(...),
    bus_id: str = Query(...),
    stop: str = Query(...)
):
    # Look up current bus position from cache/DB
    features = build_feature_vector(route_id, bus_id, stop)
    eta = model.predict(pd.DataFrame([features]))[0]

    return {
        "bus_id": bus_id,
        "route_id": route_id,
        "stop_id": stop,
        "eta_minutes": round(eta, 1),
        "confidence": classify_confidence(eta),
        "timestamp": get_utc_now()
    }

OSRM — Self-Hosted Routing (Replace Google Maps API) +

What it does: Gives us driving distances, ETAs, and routes between any two GPS coordinates — using Sri Lanka road data from OpenStreetMap. Free, self-hosted, no API keys.

# 1. Download Sri Lanka OSM data
wget https://download.geofabrik.de/asia/sri-lanka-latest.osm.pbf

# 2. Run OSRM backend (Docker)
docker run -t -v "${PWD}:/data" osrm/osrm-backend \
  osrm-extract -p /opt/car.lua /data/sri-lanka-latest.osm.pbf

docker run -t -v "${PWD}:/data" osrm/osrm-backend \
  osrm-partition /data/sri-lanka-latest.osrm

docker run -t -v "${PWD}:/data" osrm/osrm-backend \
  osrm-customize /data/sri-lanka-latest.osrm

# 3. Start the server
docker run -t -p 5000:5000 -v "${PWD}:/data" osrm/osrm-backend \
  osrm-routed --algorithm mld /data/sri-lanka-latest.osrm

Usage:

# Get road distance: Pettah → Nugegoda
curl "http://localhost:5000/route/v1/driving/79.8510,6.9355;79.8773,6.8720"

# Response includes:
#   "distance": 12340,    (metres)
#   "duration": 1680,     (seconds)
#   "geometry": "..."     (route polyline for map display)

We pre-compute all stop-to-stop distances using this and store as a lookup table.
